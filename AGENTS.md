# nervepack — AI agent manual (tool-neutral)

Host-specific wiring lives in the per-tool file (`CLAUDE.md` for Claude Code); this file is the portable core.

**Before changing any code, read `docs/ARCHITECTURE.md`** — the cheap, high-level map of features, wiring, and the
"touch X → check Y" change-impact table, so a local edit doesn't break another feature or a proven pattern.

## What this repo is

The versioned engine for a context hub that follows Pat across machines — the
machinery (hooks, crons, toggles, MCP, onboarding) plus the generic
cognition/workflow skills; domain knowledge and dev-environment setup live in the
content overlay it reads. Three audiences:

1. **You (an AI session running here)** — to pick up context and refine it.
2. **A fresh Ubuntu dev box** — clones this and bootstraps via `engine/setup/*.sh`.
3. **Other AI tools** (Cursor, Codex, etc.) — read the same skills/ and rules.

## Directory contract

**Engine vs. content overlay.** This repo (`~/Code/nervepack`) is the **engine** —
the reusable machinery plus the generic, publishable cognition/workflow skills
(`np-core-*`/`np-flow-*`; the domain/environment tiers `np-kb-*`/`np-env-*` are
content and ship in the overlay). Your **personal content** lives in a **separate overlay repo** (e.g.
`~/Code/nervepack-content`), resolved at runtime by `NP_CONTENT_DIR` (default, when
unset: the engine root, for the single-repo legacy layout). The content layers —
`wiki/` (curated synthesis pages **with their sources co-located inside**, not a
separate `sources/` dir), `memory/` (the agent-owned `episodic/`, `lessons/`
layers grouped under one dir), `archive/`,
`compact-proposals/`, `log.md`, `dashboard/data/`, your **personal**
`np-kb-*`/`np-env-*` skills (your sites, secrets, machine setup), and your
project's **visual identity** (`brand/` — logos, social card, banner, favicon,
brand guide, design language) — **do not live in the engine repo**; commit them
to the overlay. The engine repo stays free of personal data (the `pii-guard` CI
job + `np-publish-scan.py` enforce this). An optional **team** overlay may sit above the personal overlay (`team > personal > engine`); reads merge with team winning, writes still target personal. Configured via `NP_TEAM_DIR` / `~/.config/nervepack/team-dir`, gated by the `team` toggle. The value may be a comma-separated list of up to 4 team dirs, highest-precedence first (`squad,division,org` → `squad > division > org > personal > engine`); more than 4 is a hard error and the session falls back to personal-only. Topic layers (lessons/episodic/wiki) combine per the `team.merge` toggle param — `override` (default), `concatenate`, or `team-only`; skills stay override-only. `contribute` can target the team overlay (`--layer team` / 'save to the team layer'), writing via `np_team_dir`; auto-capture still writes personal.

**Brand/visual identity is content, not engine.** A specific brand is instance
identity, not reusable machinery — the engine stays identity-neutral so a forker
brings their own (the same reason the personal design-palette skill
`np-kb-branding` lives in the overlay). Only when the brand is final do the
genuinely public-facing assets (the social-preview card, a README banner/logo)
get copied into the public engine repo — GitHub needs those public to use them;
the brand guide, design language, and source stay in the overlay.

## Skill file format

Every `skills/<name>/SKILL.md` MUST start with frontmatter:

```markdown
---
name: kebab-case-matching-dirname
description: One sentence. Lead with WHAT the skill teaches, then WHEN to use it. This is what future-you reads when deciding if the skill is relevant — be specific, not generic.
---

# Human-readable title

…body…
```

Cross-link related skills with `[[skill-name]]`. Dangling links are fine —
they mark something worth writing later.

## Working protocols

### Harness language policy (bash vs Python)

The harness is deliberately **bilingual**, split on one question: *is there real
logic here, or is this just glue?*

- **Bash** — for latency-critical, low-logic glue: any script that's mostly
  sequencing other CLIs (git, jq, `np-llm.sh`, crontab). Bash's ~5–10ms cold
  start matters on these; Python's ~30–80ms does not.
- **Python (stdlib-only)** — for parsing and data-structure assembly that runs
  off the hot path (SessionEnd, cron): the evaluator's signal extraction, record
  assembly, episodic distillation, aggregation. This is where bash's footguns
  (`pipefail` turning a no-match `grep` into a hard failure, `grep -c` printing `0`
  *and* exiting non-zero, word-splitting, SIGPIPE) cost real correctness — the same
  class of bug bit the toggle CLI, the cron guards, *and* the signal extractor before
  the port. **No third-party deps** (no `pip install`, no venv) — that dependency-free
  property is the reason bash was chosen in the first place; don't trade it away.
  Tests use stdlib `unittest`, not pytest, for the same reason.
- **Single-source shared logic.** When a Python script needs a toggle decision, it
  shells out to bash `np_enabled` (`np-eval-signals.py` does this) rather than
  reimplement the resolver — the toggle resolver stays bash, owned in one place.
- **Not Go/Rust/Deno.** Go would solve both latency *and* correctness, but committing
  and distributing binaries fights the "clone and it bootstraps" ethos. Reconsider
  only if hot-path latency ever becomes the dominant constraint.

**Reference port:** `engine/setup/np-eval-signals.py` (+ `tests/evaluator/test_signals.py`)
is the proof-of-concept for the pattern — copy its shape for new Python harness code.

**Running the suite:** `bash engine/setup/tests/run-all.sh` (whole suite, zero deps,
excludes e2e); add `--with-e2e` to include the Playwright dashboard test (requires
`pip install -r engine/setup/tests/e2e/requirements.txt` + `python3 -m playwright install chromium`).
CI gates `main` on `syntax` + `regression` + `pii-guard`; `dashboard-e2e` is
informational. See `engine/setup/tests/README.md`.

**Standing rule — keep asking the question.** When you add a new harness script, or
*substantially extend* an existing one, explicitly evaluate it against this rubric
before writing — state the verdict ("staying bash: hot-path glue" / "Python: parsing
logic") in one line. Migrate bash→Python *opportunistically* (when a script bites or
you're already reworking it), never as a big-bang rewrite of working, tested code.

### Model selection policy (which model to use)

Every feature that calls `np-llm.sh` (the tool-neutral model seam) must **pin a model**
(never inherit the interactive default) and use the **cheapest model that does the job**.
The split mirrors the language policy:

- **Haiku** (`claude-haiku-4-5-20251001`) — the **default**. Single-shot
  extractive/transform passes with no tool use and bounded output: summarize a
  transcript, score a session, classify, redact, reformat. Both SessionEnd judges
  (`episodic-capture`, `np-evaluator`) use it. Keep raw input small (cap +
  extract — see [[np-kb-claude-headless-scripting]] §6) rather than reaching for a
  bigger model to chew through more tokens.
- **Sonnet** (`claude-sonnet-4-6`) — only for **agentic** passes that read/edit
  files over multiple steps and need real judgment: the episodic-maintain
  distillation (`72-`) and memory-promote (`71-`) crons. They run daily but no-op
  (no model call) on an empty inbox, so the cost lands only on days with real work.
- **Opus — not used by any nervepack feature, by policy.** No feature's quality
  ceiling justifies Opus token spend. If a task feels like it needs Opus, that's a
  signal to **decompose** it (cheap extract → cheap act), not to upgrade the model.

**Standing rule — keep asking the question** (as with language): when you add or
extend a feature that invokes a model, state the verdict in one line ("Haiku:
single-shot summarize" / "Sonnet: agentic file edits") and pin it. Default
*down*, not up. Cross-ref [[np-kb-claude-headless-scripting]] §5.

### When the user says "sync nervepack"
Invoke [[np-core-sync]]. It pulls latest, relinks skills into the host's
skill directory, re-runs the lifecycle hook installers when a fast-forward
landed engine changes (so renamed/added hooks take effect without a manual
re-onboard), and reports what changed.

### When the user says "remember this in nervepack" / "save this to nervepack"
Invoke [[np-core-contribute]]. It picks the right file, writes the update,
shows a diff, then commits and pushes. **Standing preference (set 2026-06-02):
for the private content overlay (personal skills, wiki, sources) — update + push
by default, no per-update confirmation.** (Still never force-push; on a
non-fast-forward, pull --rebase --autostash and surface conflicts.) **Engine-repo
changes follow the PR workflow below, not direct push.**

### Engine change workflow (public repo: PR → merge on green)

The engine repo is public, so substantive human/agent-authored engine changes —
code under `dashboard/`, `engine/`, and the generic `np-core-*`/`np-flow-*`
skills shipped here — go through a **pull request that merges only on green CI**,
never direct-to-main. Flow: branch off latest `origin/main` → push → `gh pr create`
→ wait for the required checks (Syntax sweep, Regression suite, Secret/PII guard;
Dashboard e2e is informational) → if the branch is behind, `gh pr update-branch`
and re-wait → `gh pr merge --squash --delete-branch` once green. Keep engine commit
messages and PR bodies **company-neutral** (feature + tech only).

**Exempt (still direct-push, bounded):** the auto-committing crons
(episodic-maintain, metrics aggregator, memory-promote, skill-maintain) — they
cannot open PRs; and the **private content overlay** (`nervepack-content`), which
has no public CI gate.

## Knowledge layer (model + precedence)

Nervepack's layers in descending authority: `skills > sources > wiki > lessons > episodic`.
(Sources are **part of the wiki layer** — source files co-located inside
`wiki/topics/<topic>/` and `wiki/concepts/<concept>/`, not a separate directory. The
`memory/` grouping is a directory convenience and does **not** change the relative
authority of episodic/lessons.)

- **Skills** — behavioral guidance ("how to act"); human-reviewed gate before write.
- **Sources** — durable, versioned technical reference co-located within the wiki layer; ingested via the protocol below.
- **Wiki** — LLM-owned synthesis pages linking skills + their co-located sources.
- **Lessons** — auto-distilled patterns from past failure→recovery *and* past successes, tagged `provenance: failure|success`; may optionally be enforced at runtime (independent of provenance).
- **Episodic** — lowest-authority working narrative; auto-captured, prunable.
- **Feature toggles** — every feature has an on/off switch; see § "Feature toggles".
- **Evaluator** — per-session scoring; feeds the dashboard and suggestion pipeline.
- **Skill maintenance** — daily budget enforcement keeps skill bodies trim.

For per-feature wiring details (hook names, cron schedules, file paths), see
`docs/ARCHITECTURE.md` and `docs/FEATURES.md`.

### Episodic layer (what it is)

Nervepack's lowest-authority layer: `skills > sources > wiki > episodic`. It records
*what was worked on*, themed by topic, and is explicitly second-class — working
narrative, may be stale, prunable.

Durable rules must NOT live here — if an episodic entry is really a rule or
preference, promote it to a skill via [[np-core-contribute]]. The maintenance
agent flags such entries in its report but does not write to `skills/` itself.

### Lessons layer (what it is)

Auto-distilled patterns from past sessions — both failure→recovery and proven
successes. Precedence: `skills > sources > wiki > lessons > episodic`. Each entry
carries a `provenance` tag (`failure` or `success`) that shapes its recall framing,
and *independently* may carry an optional `enforce` block: unlike episodic
narrative, an entry with `enforce` is **enforced at runtime** (`warn` injects at the
tool call; `ask` gates execution), regardless of which provenance it carries. Most
entries carry no `enforce` block and stay advisory-only. A proven lesson is
promoted to a `skills/np-kb-*` rule via the human-reviewed `np-core-contribute`
gate, then marked `promoted`/archived. This is the validation gate; candidate
lessons stay low-authority and reversible.

### Feature toggles (what they are)

Every Nervepack feature has an on/off switch. `engine/setup/toggles.conf` (committed) is
the manifest; `engine/setup/np-toggle-lib.sh` resolves `~/.config/nervepack/toggles.local
→ toggles.conf → default-on`. Flip via `engine/setup/nervepack-toggle.sh <feature> on|off`
(or no args for an interactive picker), or the [[np-core-toggle]] skill.

- **shared** features (memory, lessons, directive, sync) store state in the repo
  and propagate on the next sync; **local** ones (allowlist) and sub-overrides
  write to the local file. Sub-toggles (`memory.recall`) inherit their family.
- **A flag that needs its own explicit default — or just needs to be discoverable
  in the manifest — must be a *param*, not a sub-toggle.** A sub-toggle inherits
  its parent's state (an "on" parent makes any `evaluator.foo` sub-toggle resolve on)
  and never appears in `toggles.conf`, so it is invisible until someone reads the code.
  Expressing the flag as a param bakes the default into the manifest and is read via
  `np_param` with that default repeated as the fallback arg — e.g.
  `[[ "$(np_param evaluator.dashboard_serve on)" == on ]]`.
  This matters most for a **default-OFF opt-in whose parent is on**.

**Convention — every new feature MUST be toggleable.** When you add a hook, cron,
or managed config:
1. Add a `toggles.conf` row (`feature|scope|enforce|state|param`).
2. Wire enforcement: runtime/cron scripts source `np-toggle-lib.sh` and add
   `np_enabled <feature> || exit 0` (fail-open); managed config gets an install +
   a paired remove.
3. `engine/setup/nervepack-toggle.sh audit` flags hooks/cron missing a toggle family.

### Evaluator (what it is)

At session end, `engine/nervepack_engine/hooks/evaluator.py` (via `np_evaluator.evaluate()`,
toggle `evaluator.judge`) scores how much
Nervepack helped: deterministic signals (`np-eval-signals.py`) + a Haiku verdict
(score, helped, shortfalls, suggestions with confidence/auto_safe, assets_used). Records
drain into committed `dashboard/data/metrics.jsonl` via a daily aggregator. All under
the `evaluator` toggle family; all fail-open. The dashboard (P2) and **suggestion
implement/reject (P3)** consume this record; asset-usefulness flagging (P4) is still a
separate future feature. For wiring details see `docs/ARCHITECTURE.md`.

### Skill maintenance (what it is)

`engine/setup/np_skill_maintain.py` (dispatched via `engine/nervepack_engine/cli.py` as
`cli.py cron skill-maintain`, daily cron, 09:15) keeps skill bodies within budget,
deterministic-first. A detector flags any `SKILL.md` over the hard `split_kb` limit;
only then does a Sonnet model pass (via `np-llm.sh`) move overflow detail into
`skills/<name>/references/`. A deterministic gate enforces: body now under budget,
frontmatter unchanged, no links dropped. Pass → commit (+push); fail → revert.
The detector also flags when the always-loaded catalog crosses `skills.catalog_tok`
(4000 tok) — the future trigger to regroup related skills under parent-router skills.
All thresholds are tunable toggle params. For wiring details see `docs/ARCHITECTURE.md`.

## Wiki layer (sources + wiki + log)

Nervepack has two layers. **Skills** = behavioral guidance ("how to act"). The
**wiki layer** = knowledge accumulation ("what we know"), inspired by
Karpathy's LLM-wiki pattern. The two are complementary, not interchangeable.

| Layer | Lives in | Owned by | Purpose |
|---|---|---|---|
| Skills | `skills/` | User + LLM (via [[np-core-contribute]]) | Rules, conventions, plugin choices |
| Sources | `wiki/topics/<topic>/` (alongside synthesis page) | LLM via ingest protocol below | Curated technical reference docs |
| Wiki | `wiki/topics/<topic>/` (synthesis) · `wiki/concepts/` | LLM (regenerated as inputs change) | Synthesis pages cross-linking skills + sources |
| Log | `log.md` | All writers (append-only) | Chronological audit |

### Source curation rule (what qualifies)

A source file at `wiki/topics/<topic>/<name>.md` (with `kind: reference` frontmatter) MUST be:

- **Official technical reference** — language books, API docs, RFCs, PEPs,
  spec documents, official tutorials.
- **Version-pinned** — frontmatter records exact version / release / SHA.
- **Trimmed to consulted scope** — chapter, section, or API surface you
  actually consult. Do not mirror entire doc trees. Pointers + targeted
  excerpts win over full mirrors every time.

A source file MUST NOT be:

- Vendor blog posts, "10 best practices" articles, conference talks.
- Stack Overflow answers, GitHub issues, aggregator summaries.
- Anything written to sell, persuade, or rank — **no marketing fluff.**
- A duplicate of [context7](https://github.com/upstash/context7) coverage
  unless durability or version-pinning is the explicit goal (see boundary
  below).

### Source frontmatter

```yaml
---
name: <kebab-case-matching-filename>
source_url: <canonical upstream URL>
version: "<release tag, edition, SHA, or RFC revision>"
captured_date: YYYY-MM-DD
scope: "<which chapters/sections/APIs are covered>"
license: "<license of the upstream material>"
topic: <matches the parent topic folder, e.g. rust, python, aws>
kind: reference
---
```

Synthesis-vs-source is told by `kind:`, not by location or filename. A synthesis page carries `kind: topic` or `kind: concept`; every co-located reference file carries `kind: reference`. `[[name]]` wikilinks resolve by name, not path, so moving files does not break links.

### Boundary: wiki/topics sources vs context7

- `wiki/topics/<topic>/` sources = **durable, versioned, cross-linkable**. Use for things you
  consult repeatedly and want to annotate in the wiki layer.
- `context7` (Claude plugin) = **live, just-in-time**. Use for "what's the
  latest API for X right now." Cannot be cross-linked or built upon.

If the question is repeat-volume *and* benefits from a fixed version,
ingest to `wiki/topics/<topic>/`. If it's one-off or you want bleeding-edge, use
context7. Don't duplicate.

### Defer-first protocol (query operation)

When answering a question:

1. Check `INDEX.md` for relevant skills.
2. Check `wiki/topics/` and `wiki/concepts/` for a synthesis page that covers the topic.
3. Check `wiki/topics/<topic>/` for co-located reference sources (`kind: reference`).
4. If a source covers the question, **read it first**, answer from it,
   and cite the source by file path. Note its `version` if relevant.
5. If no source exists but the topic feels source-worthy, propose an
   ingest (see protocol below).
6. If the question is bleeding-edge / version-sensitive and no source
   covers it, fall back to context7.
7. Only after all of the above, answer from model knowledge — and say so.

### Ingest protocol — adding a source

Triggered when: the user provides a URL/excerpt to ingest; a query hits a
knowledge gap and the LLM proposes filling it; or [[np-core-contribute]]
routes a learning to a wiki source (`wiki/topics/<topic>/<name>.md`).

**Limits.** Soft cap: 100 total sources. Hard cap: 300 total sources.
Override with explicit user permission only.

```bash
TOTAL=$(find "$NP_CONTENT_DIR/wiki/topics" -name '*.md' ! -name '<topic>.md' | wc -l)
PARENT=$(find "$NP_CONTENT_DIR/wiki/topics/<topic>" -name '*.md' ! -name '<topic>.md' 2>/dev/null | wc -l)
```

When proposing a new source, the LLM MUST surface — via `AskUserQuestion`
or equivalent prompt — the following so the user can choose with full
context:

> About to ingest **`<source-name>`** into `wiki/topics/<topic>/`.
>
> - Sources in `wiki/topics/<topic>/`: **N** (least-referenced: `<name>` with K
>   backlinks from wiki/)
> - Total sources in nervepack: **M / 100 soft / 300 hard**
>
> Choose:
> - **Add** — append the new source. (Disabled if total ≥ 300 without override.)
> - **Rotate** — archive `<least-referenced-source>` to `archive/sources/`,
>   then add the new source to `wiki/topics/<topic>/`. (Recommended once total ≥ 100.)
> - **Skip** (Other:) — don't ingest; explain why or redirect.

Computing "least referenced": count `[[<source-name>]]` backlinks from
`wiki/topics/<topic>/` and `wiki/concepts/` plus any inbound mentions in `skills/`. Ties broken by oldest
`captured_date`.

After adding: append a `## [YYYY-MM-DD] ingest | <name>` entry to
`log.md`, update the `wiki/topics/<topic>/<topic>.md` synthesis page
to link the new source (create it with `kind: topic` if absent), and commit (skill-style: `source(<topic>): add <name>`).

### Wiki synthesis pages (topic/concept pages)

**Delineation rule (decidable):** Does this page synthesize a body of curated reference sources?

- **Yes → `topic`**: create a folder `wiki/topics/<topic>/`; place the synthesis page at `wiki/topics/<topic>/<topic>.md` (frontmatter `kind: topic`); co-locate reference sources inside the same folder (frontmatter `kind: reference` + `topic:`).
- **No → `concept`**: place the page at `wiki/concepts/<name>.md` (frontmatter `kind: concept`); no co-located sources.

This replaces the old thing-vs-idea entity/concept split. A topic folder with sources but no `kind: topic` synthesis page is allowed — the lint pass flags it as a synthesis gap.

Frontmatter for a **topic synthesis page**:

```yaml
---
name: <kebab-case>
kind: topic
last_updated: YYYY-MM-DD
sources: ["<source-name>", ...]    # which source files in this folder the page draws from
---
```

Frontmatter for a **concept synthesis page**:

```yaml
---
name: <kebab-case>
kind: concept
last_updated: YYYY-MM-DD
---
```

A wiki page is regenerated (or surgically updated) when:

- A new source is ingested on the same topic.
- A skill that the page cross-links changes materially.
- The lint pass flags it as stale or contradictory.

Wiki pages should cite both skills (`[[skill-name]]`) and sources
(`[[source-name]]`). `[[name]]` wikilinks resolve by name, not path — moving files doesn't break links. Dangling links are fine — they signal future work.

### Lint operation (health check)

Periodically (or on user request "lint nervepack"), surface:

- **Stale sources** — `captured_date` > 6 months old AND upstream version
  has changed.
- **Orphan sources** — `kind: reference` files with no inbound `[[links]]` from `wiki/` or `skills/`.
- **Orphan wiki pages** — no inbound links and no sources listed.
- **Contradictions** — same fact stated differently across pages.
- **Approaching limits** — total sources ≥ 90 (10 from soft cap) or ≥ 270
  (10 from hard cap).
- **Synthesis gaps** — a `wiki/topics/<topic>/` folder contains source files but has no `kind: topic` synthesis page (`<topic>.md`).

Don't auto-fix. Surface candidates and let the user decide.

## When adding a new skill
1. **First read `INDEX.md`** — find existing skills with overlapping
   topic/triggers. Extend the existing one if there's any meaningful
   overlap. This is how disparate sessions avoid creating duplicates.
2. `mkdir skills/<name> && ${EDITOR:-nano} skills/<name>/SKILL.md`
3. Add `./skills/<name>` to `skills` array in `.claude-plugin/plugin.json`
4. Run `~/Code/nervepack/engine/setup/30-link-skills.sh` — symlinks into the host's
   skill directory (e.g. `~/.claude/skills/` for Claude Code) AND regenerates `INDEX.md`.
5. Commit with subject `skill(<name>): <one-line summary>`

If the addition is a *feature* (a hook, cron job, or managed config — not just a
knowledge skill), it MUST also get a toggle — see § "Feature toggles".

## When porting an external/community skill into nervepack
Don't drop a foreign skill in verbatim — adapt it so it belongs:
1. **Re-tier the name** to `np-<tier>-<name>` (`core`/`kb`/`env`/`flow`) so it joins
   the namespace and the session directive's routing. Dirname == frontmatter
   `name`.
2. **Keep the original bare `/command` as a *trigger phrase* in the description**
   (not the skill name) — that's how `/handoff` routes to `np-core-handoff` and
   `/recall` to `np-core-recall`. The model maps the bare command to the prefixed
   skill.
3. **Cross-link its nervepack family** with `[[…]]` and say what it is *not* (which
   sibling to use instead) so it doesn't read as a duplicate of an existing layer.
4. **Wire it to repo conventions** — temp-dir not repo for ephemeral output, redaction
   via [[np-env-secrets-refresh]], reference-don't-duplicate, point "suggested skills"
   at real `np-*` skills.
5. **Verify before deploying** (a disciplined skill-authoring/verification pass — e.g.
   superpowers `writing-skills` if your host has it): one subagent application
   test that an agent given only the skill produces compliant output. Then the standard
   add-skill steps (plugin.json → `30-link-skills.sh` → commit).

## When removing a skill
1. `mv skills/<name> archive/<name>` (preserve history; never delete)
2. Append a row to `archive/MANIFEST.md` (retired date, reason, replaced-by)
3. Remove entry from `.claude-plugin/plugin.json` `skills` array
4. Run `30-link-skills.sh` — its prune pass removes the dangling symlink
   and regenerates `INDEX.md`.
5. Commit with subject `skill(<name>): archive — <reason>`

## When you find a compact-proposal awaiting review
`compact-proposals/<date>.md` contains merge/split suggestions from the
weekly-compact agent. Read it, decide for each proposal, then either:
- Apply it (do the merge/split as the proposal describes, archive losers,
  delete the proposal file), or
- Reject it (delete the proposal entry, add a one-line note in
  `compact-proposals/REJECTED.md` so the agent doesn't re-propose).

## When another AI session may be working in this repo concurrently
`~/Code/nervepack` is a **single working tree with one git HEAD**. If two sessions —
or a session plus a cron/cloud agent — operate here at once, they WILL collide.
Observed failure modes (all real):
- `git add -A` / `git commit -am` in one session **sweeps the other session's
  untracked files into your commit**. **Always stage explicit paths**
  (`git add <file> …`) — never `-A`, `.`, or `-am`. **And commit explicit paths too**
  (`git commit -m "…" -- <file> …`): a *bare* `git commit` after an explicit `git add`
  still commits the **whole index**, so it captures whatever else another session has
  staged — the `git add` scope is lost the moment a pathspec-less `commit` runs against
  the shared index. (Observed live: the metrics cron's bare `commit` swept a concurrent
  session's staged WIP into a mislabeled `evaluator(metrics)` commit — issue #11. Every
  auto-committing cron/agent here now path-limits its `commit`.) Better still where the
  work is detachable, commit from an isolated `git worktree` off the committed base (the
  suggestion-implement job already does this).
- A `git reset`/rebase in one session **orphans the other's commits** off the
  branch tip. They survive in the object store + `git reflog` but vanish from the
  branch, so a naïve `push` would silently drop them.

Operating rules when concurrency is possible:
1. **Lock work to the remote early.** Once you've verified the exact commit that
   contains everything, fast-forward-push that *pinned SHA* to `origin/main`
   (`git push origin <sha>:refs/heads/main`). Origin is then immune to any local
   reset the other session does next.
2. **Tag a backup** (`git tag <name> <sha>`) before any reset/rebase/force op.
3. **Recover orphans** from `git reflog` (find the lost SHA, `git branch`/
   `cherry-pick` it) — verify content with `git show <sha>:<path>` before trusting.
4. **Realign before committing** after a known concurrent run: `git fetch` and
   treat `origin/main` as the source of truth, not the local branch tip.

## What NOT to do

- **Don't write durable facts into session-scoped memory stores.** That
  store is session-scoped. Promote anything worth keeping into a skill here.
- **Don't bypass the symlinks.** Skills live in `~/Code/nervepack/skills/` — the
  symlinks in the host's skill directory (e.g. `~/.claude/skills/`) are the delivery
  mechanism, not the source.
- **Don't force-push.** Nervepack is small, conflicts should be rare; if one
  appears, merge with `--no-edit` or surface to the user.

## Commit conventions

```
skill(<name>):    <change>    # adding/editing a skill
source(<topic>):  <change>    # adding/editing/rotating a source
wiki(<area>):     <change>    # adding/editing a wiki topic/concept page
episodic(<topic>): <change>    # editing the episodic layer (usually the maintenance agent)
log:              <change>    # log.md edits (rare — should be append-only)
setup(<step>):    <change>    # editing a bootstrap script
manual:           <change>    # editing AGENTS.md, CLAUDE.md, or README.md
agent(<name>):    <change>    # editing scheduled-agent prompts
```

Trailers: **do not add LLM attribution** — no `Co-Authored-By: <AI>` trailer, no
"Generated with …" line, regardless of model/IDE. Author as the repo's configured git identity. See
[[np-kb-coding-rules]] § 6. (Applies to every agent + skill in this repo,
including the auto-committing cron agents below.)
